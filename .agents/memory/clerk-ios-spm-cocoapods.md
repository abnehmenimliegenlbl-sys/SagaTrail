---
name: Clerk iOS SPM + CocoaPods build crash
description: Native iOS build (EAS/Expo Launch) fails with "undefined method package_product_dependencies for nil:NilClass" in react-native's spm.rb when Clerk is installed.
---

`@clerk/expo`'s iOS podspec pulls in the native Clerk SDK (`ClerkKit`/`ClerkKitUI`) via React Native's `spm_dependency` CocoaPods-SPM bridge (RN >= 0.75). If the Podfile doesn't force dynamic framework linking, `pod install` crashes in `react-native/scripts/cocoapods/spm.rb` (`add_spm_to_target`) with `undefined method 'package_product_dependencies' for nil:NilClass` — the generated Xcode target for the SPM-consuming pod can't be resolved.

**Why:** RN's CocoaPods-SPM integration only reliably resolves targets when the whole iOS project uses `use_frameworks! :linkage => :dynamic`. Without it, static-linking pods (the RN/Expo managed-workflow default) leave the SPM package's target wiring incomplete.

**How to apply:** In an Expo managed-workflow project (no committed `ios/` folder — Podfile is generated at build time), fix it via config plugin, not by hand-editing a Podfile: install `expo-build-properties` (pin to the version matching the installed Expo SDK, e.g. `~1.0.10` for SDK 54 — `pnpm add` alone may resolve an incompatible major/latest version) and add to `app.json` plugins:
```json
["expo-build-properties", { "ios": { "useFrameworks": "dynamic" } }]
```
This emits the required `use_frameworks! :linkage => :dynamic` in the generated Podfile.

**This alone is NOT sufficient for `@clerk/expo` specifically.** `ClerkExpo.podspec` also sets `s.static_framework = true` on itself, which overrides/conflicts with the project-wide dynamic linkage for that one pod and causes CocoaPods/RN's SPM bridge to lose track of the `ClerkExpo` pod's own Xcode target (it never appears in the `pod install` target list at all) — same crash, same `spm.rb:80` site, even with dynamic linkage already active project-wide. Confirmed root cause via a full `pod install` log: "Adding SPM dependency on product" fires right before the nil crash, and grepping the entire log shows no `ClerkExpo` line in the installed-pods list.

Fix: patch out `s.static_framework = true` from `@clerk/expo`'s `ios/ClerkExpo.podspec` so it inherits the project's dynamic linkage like every other pod. Since this lives in `node_modules` (not the repo), use pnpm's native patch feature — `pnpm patch @clerk/expo@<version>`, edit the podspec in the temp checkout, `pnpm patch-commit <path>` — which records the patch under `patches/` and `patchedDependencies` in `pnpm-workspace.yaml` so it reapplies on every `pnpm install`, including in Expo Launch's remote build. Do not guard-clause `spm.rb` itself (`return unless target`) as an alternative — that just silently skips linking `ClerkKit`/`ClerkKitUI`, likely trading the install crash for a later missing-symbol crash.

**Follow-on crash after the podspec patch (same investigation, different failure):** once `ClerkExpo`'s target resolves correctly, `pod install` can still fail with `The 'Pods-<Target>' target has transitive dependencies that include statically linked binaries: (ExpoModulesCore)`. This is unrelated to Clerk directly — Expo's own autolinking always force-disables dynamic frameworks for a fixed set of "core" pods (seen: `ExpoModulesCore`, `Expo`, `ReactAppDependencyProvider`, `expo-dev-menu`, logged as `[Expo] Disabling USE_FRAMEWORKS for modules ...`), which conflicts with a project-wide `use_frameworks! :linkage => :dynamic` (itself required for Clerk's SPM dependency above). CocoaPods refuses to mix a static binary into a dynamic-framework dependency graph.

**True root cause (found by reading the actual Expo source):** The "Disabling USE_FRAMEWORKS" operation is NOT done inside a Podfile `pre_install` block — it is done inside `expo-modules-autolinking`'s `installer.rb` (located at `scripts/ios/cocoapods/installer.rb`), which monkey-patches `Pod::Installer#run_podfile_pre_install_hooks` using `define_method`. The patched method: (1) calls the *original* CocoaPods method (which runs ALL Podfile `pre_install` blocks), THEN (2) applies the static_library downgrade. This means no Podfile `pre_install` block — regardless of registration order — can win, because Expo's downgrade always runs AFTER every Podfile hook.

**Correct fix:** Inject a Ruby snippet at the end of the Podfile (after `use_expo_modules!` has been called and `installer.rb` has been `require`d) that does another `define_method(:run_podfile_pre_install_hooks)` override on `Pod::Installer`. At Podfile evaluation time, `instance_method(:run_podfile_pre_install_hooks)` captures Expo's version; the new `define_method` replaces it. The new version calls Expo's version (which calls the real CocoaPods method + applies the static_library downgrade), then applies a final `def pod.build_type; Pod::BuildType.static_framework; end` singleton override on the affected pods. Build type must be `static_framework` (not `static_library`) — a static framework is compatible with `use_frameworks! :linkage => :dynamic`; a static library is not.

The condition that triggers Expo's downgrade (from `should_disable_use_frameworks_for_core_expo_pods?`): `RCT_USE_PREBUILT_RNCORE == '1'` AND `USE_FRAMEWORKS` env var set. Both are true on EAS for Expo 54 builds (confirmed via the log: "Using React Native Core ... prebuilt versions" + `use_frameworks! :linkage => :dynamic` from expo-build-properties).

Since the Podfile is regenerated on every prebuild (managed workflow, no committed `ios/`), inject this via a custom local Expo config plugin using `withDangerousMod("ios", ...)` appending at end of file, referenced in `app.json`'s `plugins` array as `./plugins/withStaticFrameworksFix.js`. Import config-plugin APIs via `expo/config-plugins`, not `@expo/config-plugins`. Verified via `npx expo prebuild --platform ios --no-install` + grep that the `define_method(:run_podfile_pre_install_hooks)` snippet appears at end of Podfile, after target block. Prebuild also touches `package.json` (adds run scripts + duplicate deps) — diff and revert afterward.

**Third crash, one step earlier — `expo config --json exited with non-zero code: 1` during EAS's "Resolving build config..." step, before pod install even starts:** caused by a custom local config plugin requiring `@expo/config-plugins` directly. That package is only a *transitive* dependency (pulled in via `expo`), not declared in the leaf package's `package.json`, so under pnpm's strict/isolated node_modules it can resolve "by accident" in one environment (e.g. hoisted at the monorepo root for unrelated reasons) while failing to resolve in a fresh install elsewhere (e.g. the EAS build machine) — `npx expo config --json` can even exit 0 locally while the exact same require fails remotely, so a clean local repro does not rule this out. EAS's own error output for this step is extremely sparse (no stack trace surfaces even in the full log), so treat "config resolution fails only on EAS" as this exact class of bug first.

Fix: in any custom Expo config plugin, never `require("@expo/config-plugins")` directly. Use `require("expo/config-plugins")` instead — `expo` is always a real declared dependency of an Expo project, and it re-exports the same config-plugin APIs (`withDangerousMod`, etc.) at that subpath specifically so custom plugins don't need their own separate, easy-to-miss dependency declaration.

**Ruby-Namespacing-Falle in Podfiles (vierter Crash):** Die Podfile wird im DSL-Kontext `Pod::Podfile` evaluiert. Ein `module Pod; class Installer ... end; end` im Podfile-Text legt darum eine NEUE leere Klasse `Pod::Podfile::Pod::Installer` an, statt die echte `::Pod::Installer` zu oeffnen — Fehler: "undefined method `run_podfile_pre_install_hooks' for class `Pod::Podfile::Pod::Installer'". In Podfile-injiziertem Ruby-Code IMMER absolute Konstantenpfade verwenden: `::Pod::Installer.class_eval do ... end` und `::Pod::BuildType`.

**ENDGUELTIGE ERKENNTNIS (fuenfter Crash — static_framework hilft NICHT):** CocoaPods' TargetValidator hat ZWEI Pruefungen in `verify_no_static_framework_transitive_dependencies`. Die zweite lehnt JEDES statische Target ab, von dem ein dynamisches Pod abhaengt: `static_deps = dynamic_pod_targets.flat_map(&:recursive_dependent_targets).uniq.select(&:build_as_static?)` — `build_as_static?` ist auch fuer static_framework true. Der gesamte Ansatz "herabgestufte Pods nachtraeglich auf static_framework upgraden" kann diese Validierung PRINZIPIELL NIE bestehen (der Installer-Override selbst funktionierte einwandfrei — Podfile evaluierte fehlerfrei, Expos Downgrade lief durch unseren Wrapper).

**Funktionierende Loesung:** Expos Downgrade gar nicht ausloesen: `ENV['RCT_USE_PREBUILT_RNCORE'] = '0'` als ERSTE Zeile der Podfile injizieren (Config-Plugin, withDangerousMod, prepend statt append). installer.rb prueft `return false if ENV['RCT_USE_PREBUILT_RNCORE'] != '1'` — Downgrade entfaellt, alle Pods bleiben dynamisch, Validierung besteht, Clerk-SPM funktioniert. React Native wird dann aus dem Quellcode gebaut (laengere EAS-Buildzeit, wie vor Expo 54 Standard). Die ENV-Zuweisung MUSS vor `use_react_native!`/`use_expo_modules!` evaluiert werden — deshalb Anfang, nicht Ende der Podfile.

**KORREKTUR DER GRUNDANNAHME (sechster Crash — die "dynamic"-Praemisse war falsch):** Auch mit RCT_USE_PREBUILT_RNCORE=0 schlug die Validierung fehl, obwohl Expos Downgrade-Hook nachweislich NICHT lief (keine "Disabling USE_FRAMEWORKS"-Logzeile). Grund: `ExpoModulesCore.podspec` setzt selbst hart `s.static_framework = true` — unabhaengig von jeder Downgrade-Logik. Unter `use_frameworks! :linkage => :dynamic` ist damit IMMER ein statisches Target im dynamischen Graphen -> CocoaPods TargetValidator lehnt ab. Zusaetzlich: Clerks ORIGINAL-Podspec (`ClerkExpo.podspec`) setzt ebenfalls `s.static_framework = true` — Clerk ist vom Hersteller fuer STATISCHES Linking ausgelegt; die Annahme "Clerk braucht dynamic" stammte aus einer Fehlinterpretation des fruehen spm.rb-Crashes.

**Finale Loesung:** `expo-build-properties` auf `"useFrameworks": "static"` stellen. Dann sind alle Pods konsistent statisch, `dynamic_pod_targets` ist leer, die Validator-Pruefung greift gar nicht. Der Custom-Config-Plugin (RCT_USE_PREBUILT_RNCORE-Injection) wurde entfernt — prebuilt RN Core ist mit static der normale, unterstuetzte Pfad. Der pnpm-Patch am Clerk-Podspec (static_framework entfernt) bleibt harmlos bestehen: ohne explizites static_framework erbt der Pod die projektweite statische Verlinkung, identisch zum Upstream-Verhalten.
