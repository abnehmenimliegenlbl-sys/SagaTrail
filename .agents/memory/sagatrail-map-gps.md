# swisstopo map + live GPS (live hike)

## Map rendering
- Base tiles are **Carto Voyager** (`basemaps.cartocdn.com/rastertiles/voyager`, PNG) plus a **Waymarked Trails hiking overlay** (`tile.waymarkedtrails.org/hiking`, transparent PNG, must sit over a base layer) via Leaflet inside a **WebView on native** and an **iframe `srcDoc` on web** — platform split via `SwisstopoMap.tsx` / `SwisstopoMap.web.tsx`, shared HTML from `swisstopoMapHtml.ts`. No API key needed. Both the live map AND the offline-downloaded tile corridor (`offlineTiles.ts`) use Carto Voyager now — this replaced the amtliche swisstopo WMTS raster (`ch.swisstopo.pixelkarte-farbe`) everywhere in map rendering, which the user found too visually busy/cluttered. swisstopo itself is still used only for SAC-difficulty derivation (`swisstopoHiking.ts`), a completely separate, non-visual use. This is the deliberate alternative to `react-native-maps` (which the project avoids) and to the SVG `RouteMap` (kept only as a coordinate-less fallback).
- **Client-side direct Overpass calls from a WebView/mobile screen are unreliable** (overpass-api.de returned 504s repeatedly, kumi.systems mirror hung) — any new OSM-derived overlay (e.g. aerialways/Seilbahnen) must be fetched **server-side** through the existing `overpass.ts` retry/mirror machinery and exposed via a new API route, then passed into `SwisstopoMap` as a plain prop (bbox computed client-side via a `geo.ts` helper around the route geometry). Never call Overpass directly from the mobile client.
- **The offline-downloaded tile corridor (`lib/offlineTiles.ts`) is a separate download path from the live map's tile layer** — changing the live basemap URL in `swisstopoMapHtml.ts` does NOT change what gets downloaded for offline use. Both must be updated together (same tile URL/format) or offline hikes silently keep showing the old basemap. The Waymarked Trails hiking overlay is deliberately NOT downloaded offline (extra tile set, overlay-only) — it just fails to load offline and resumes online.

## Route line (the drawn track)
- The real route path lives ONLY in `external_routes.geometry` (server, ~80-pt downsampled OSM track). It is NOT in the curated seed (`constants/routes.ts` seed routes carry only a single `coordinates` start point). To show a route line you must plumb geometry end-to-end: OpenAPI `CatalogRoute.geometry` (optional) -> codegen -> mobile `HikingRoute.geometry?: number[][]` -> `SwisstopoMap` props -> `buildSwisstopoHtml` `L.polyline` + start/Ziel markers + `fitBounds`.
- **Why:** the server already computed geometry, but `cantons.ts`/`catalog.ts` `toRoute()` silently dropped it and the map only drew a start marker, so users saw no route and asked "how am I navigated?". Keep geometry OPTIONAL in the contract so seed/catalog routes without it still validate (they fall back to a single start marker).
- **Resolve the live hike's route by the SELECTED route id, not from the saga id.** Dynamic OSM routes store their own OSM id in `sagaId` (see `routeService` — `sagaId: r.id`), NOT the curated saga's id. So `getRouteBySaga(sagaId)` on the hike/saga screens misses the OSM route and falls back to a geometry-less seed route → only the start marker draws. Thread the real `routeId` through navigation (route -> saga -> hike as a query param) and resolve `getRoute(routeId) ?? getRouteBySaga(id)`. **Why:** the route↔saga link is nearest-saga (many routes map to one saga), so a saga id cannot uniquely recover the chosen route.
- **Web preview blocks geolocation** (embedded iframe has no `allow="geolocation"`), so `locState` becomes `simulated` and there is no real position. To keep the map alive there, show an interpolated marker along `route.geometry` by `distance/totalKm` when not `granted`; real GPS only appears on device.
- There is intentionally **no turn-by-turn navigation**: the app draws the full route line + your live GPS dot and advances narration by distance — it is route visualization, not a navigator.
- Live position is pushed **without reloading tiles**: the HTML exposes `window.sttSetPosition(lat,lng)`; native calls it via `injectJavaScript`, web via `iframe.contentWindow.sttSetPosition`.
- **Define `window.sttSetPosition` in an early `<head>` script that BUFFERS the position, not at the bottom after the Leaflet `<script>`.** On iOS `react-native-webview`, `onLoadEnd` can fire (and the position injection run) before the bottom script defines `sttSetPosition`; the `sttSetPosition && …` guard then silently no-ops, and because a stationary device's position never changes the effect never retries → route line draws but the live marker never appears. The buffered value is applied by a `window.__sttApply` defined in the main script once the map is built. **Why:** this was the exact "immer noch kein marker" bug — a one-shot injection lost to a load-order race, invisible from types and from the fact that the route itself rendered fine.
- A stationary device never fires `watchPositionAsync` when `distanceInterval` is set, so also request one immediate `getCurrentPositionAsync` after permission is granted, or a user testing at home sees no fix at all.
- **Reset `ready=false` whenever the HTML string changes** (center/label), otherwise a reloaded document silently drops the initial position injection until the next GPS fix.

## GPS / simulation state machine (app/hike/[id].tsx)
- States: `idle` -> `granted` | `denied` | `simulated`. Native uses `Location.watchPositionAsync`; web uses `navigator.geolocation.watchPosition` (falls back to `simulated` on error/unavailable).
- **Only run the simulated-timer fallback in explicit fallback states** (`denied`/`simulated`), never in `idle` — otherwise a timeout fires during the permission handshake and fabricates progress before real GPS is confirmed.
- Real distance (haversine, `lib/geo.ts`) drives chapter advance when `granted`; noise <0.003 km and jumps >0.5 km are dropped.
- **Decision points must gate on `chosenOptionIndex == null`**, and narration must be de-duped with a `lastNarratedRef`. The narration effect depends on `chapters`, so `chooseOption` mutating a chapter re-runs it on the same index; without these guards it re-locks the decision and re-reads the chapter aloud.

**Why:** these three were the exact issues an architect review caught; they are subtle and not visible from types or a clean typecheck.
